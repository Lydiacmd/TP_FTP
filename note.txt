Q7 / Traitement côté client
    - Construit la request_t avec req.type = GET et le nom du fichier
    - L'envoie au serveur avec Rio_writen
    - Reçoit la response_t du serveur
    - Si rep.code == -1 → affiche un message d'erreur
    - Si succès → reçoit le fichier avec Rio_readn, le sauvegarde dans CLIENT_DIR
    - Affiche les stats : bytes reçus, temps, vitesse en Kbytes/s

Q9 /  Multi-requêtes par connexion :

Q10 /  Reprise après crash client 
    - Le client écrit dans un fichier temporaire (TRANSFER_TMP) l'avancement octet par octet à chaque bloc reçu
    - Au redémarrage, il lit ce fichier et met req.offset à la valeur déjà reçue
    - Le serveur fait un lseek(fd, req.offset, SEEK_SET) pour sauter la partie déjà envoyée
    - À la fin du transfert, TRANSFER_TMP est supprimé


Q11 / Achitecture globale

Q12 /  Interconnexion maître-esclaves
Au démarrage, le maître se connecte à chaque esclave pour vérifier qu'ils sont up et stocke leurs infos (ip + port) dans un tableau slaves[NB_SLAVES].

Q13 / Adaptation de la connexion client
Le protocole de connexion a été modifié en 3 étapes : 
le client se connecte d'abord au maître (port 2121), reçoit les infos d'un esclave (slave_info_t), 
ferme la connexion avec le maître, puis se connecte directement à l'esclave.
 Ensuite il suit le protocole normal (get/bye).


Q14 / Protocole de reconnexion en cas de panne 
- Quand l'esclave plante, Rio_readn va retourner une erreur côté client. À ce moment-là le client doit :
        Se reconnecter au maître
        Recevoir les infos d'un nouvel esclave
        Se reconnecter à ce nouvel esclave
        Reprendre le transfert depuis l'offset sauvegardé

Cote Client : 
    - la valeur retournée  par rio_readn est ≤ 0   --->  le client détecte que l'esclave est tombé --> Il ferme la connexion
    - se reconnecte au maître pour obtenir les infos d'un esclave fonctionnel -->  se reconnecte à ce nouvel esclave
    - renvoie la requête GET avec l'offset correspondant aux octets déjà reçus
    -  Le transfert reprend donc depuis l'endroit où il s'est interrompu.

Cote maitre : 
    - avant de rediriger un client vers un esclave ->  il teste si l'esclave est up en tentant une connexion avec open_clientfd (minuscule : car la gestion d'erreur nous aide)
    -  Si l'esclave est down, il passe au suivant dans l'ordre round-robin.
    - // cas tout les esclave sont down au meme temps // Si aucun esclave n'est disponible, il envoie une slave_info_t avec port = -1 pour signaler l'erreur au client // Quitte propremeent au lieu de planter //.
